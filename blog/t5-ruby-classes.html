When I first started diving into Ruby, I often heard of the terms "classes", "objects", and "instances". I knew that in object oriented programming, just about everything can be considered an object, but I had no concrete definition of what an object is when writing code, and what it can do. In an attempt to demystify the concept of the Class as well as tools such as the instance variable and instance method in Ruby, let's take a closer look at what an object is.

Suppose we are writing a program that will display your collection of video game consoles. We know that we will be dealing with game consoles, and will have to display different information for each one such as the manufacturer, the number of controllers supported, the year it was released, etc. We can think of this console as an object, or a thing that can have different properties. Now, anyone not exposed to class design might jump at the opportunity and start to translate these into variables: 

console_1_name = "xbox"
console_1_year = 2001
coneole_1_controllers = 4

console_2_name = "playstation2"
console_2_year = 2000
coneole_2_controllers = 2

Now, you can quickly see how if we were talking about types of clothing, or cars, that we will end up with a plethora of separate variable names that point to each object we want to reference. This is where the power of the object in object oriented programming comes in, and to define what an object is, we need to think in terms of a class. Take a look at the variables and you'll notice they all have one thing in common: "console" is in the name of each of these variables. Let's define our console class. Think of our class as a template that defines what properties they will have upon their creation and what type of behaviors they will able to do at that particular time.

class Console

  def initialize(name, manufacturer, year, controllers)

    @name = name
    @manufacturer = manufacturer
    @year = year
    @controllers = controllers

  end
  
end

Here, we've defined our Console class which can spawn our objects. The initialize method you see here is allowing us to pass in the name, manufacturer, year, and controllers as arguments to describe what our new object will be like. We call this object it will create as an instance of class Console. 

xbox = Console.new("Xbox", "Microsoft", 2001, 4)

ps2 = Console.new("Playstation 2", "Sony", 2000, 2)

#<Console:0x000000030b82b0 @name="Xbox", @manufacturer="Microsoft", @year=2001, @controllers=4>

#<Console:0x000000030a9620 @name="Playstation 2", @manufacturer="Sony", @year=2000, @controllers=2>

You might be wondering how we can have one defined set of variables of @name, @year, etc. but have multiple objects, each with its own properties. That is because these are instance variables, denoted by the '@' symbol, which designates that a name property only belongs to an instance of the class Console. In other words, a Console does not have a name, manufacturer, and year, but an instance of Console does! This is an important distinction.

Now that we stored a few properties of our object using instance variables, we can define some behavior using instance methods! Let's write a method that simply outputs the properties we want. There is a shortcut to this using attr_reader method, but let's write it out so we can see what is happening.

class Console

  def initialize(name, manufacturer, year, controllers)

    @name = name
    @manufacturer = manufacturer
    @year = year
    @controllers = controllers

  end
    
  def name
    @name
  end
  
  def manufacturer
    @manufacturer
  end
  
  def year
    @year
  end

  def controllers
    @controllers
  end
  
end

We've created a method to return the @name when we call the name method. Same for the others. Let's see it in action:

xbox.name

=> "Xbox"

xbox.manufacturer

=> "Microsoft"

xbox.year

=> 2001

xbox.controllers

=> 4

When we type in xbox.name, the name method gets called on to the xbox object, which already has its own instance variable of @name which stored "xbox". This is how our objects can contain its own unique properties while still belonging to a more generalized class. It's helpful to think of the object as just "one example of", or "type of" Console.
